Core Features

A cognitive stack for
autonomous agents

Every component works in concert — transforming raw interactions into durable, evolving expertise without human-labeled ground truth.

🧠
Failure-Driven Learning

LLM-as-a-judge detects hallucinations and auto-generates Failure Notes at strength 1.0. The agent never repeats the same mistake — corrections surface as the #1 retrieval result forever.

⏳
Ebbinghaus Memory Decay

Memories fade via S(t) = S₀·e^(−λt). Retrieved memories reinforce; idle ones decay below the 0.17 suppression threshold and are excluded from all future searches.

⚡
Semantic Caching

LRU cache with cosine similarity ≥ 0.92 detects paraphrased queries. Returns cached answers instantly, dropping latency from ~3000ms to ~2ms — no LLM call needed.

🕸
GraphRAG + Vector RAG

Unified memory combining cosine similarity with NetworkX graph traversal. Captures logical neighbors and relational reasoning that pure vector search misses.

🔌
Extension Wrapper API

The entire pipeline bundles into a drop-in RAGMemoryExtension with auto-save/load brain states. Ingest PDFs and query in three lines. No external database required.

📦
Pickle Persistence

Memory nodes, semantic cache LRU lists, and GraphRAG edges serialize into a single compressed byte-stream via Python pickle. Zero infrastructure overhead.

System Architecture

The FDL correction loop

A strict System-1 / System-2 duality governs every query. Fast retrieval first, deep correction only when the judge fires.

01

System-1 — Fast Retrieval

Cosine-similarity search over all non-suppressed memory nodes returns top-k results. A provisional answer is generated in milliseconds.

all-MiniLM-L6-v2 · cosine similarity
02

Adjudication — LLMJudge

An independent judge evaluates faithfulness of the provisional answer against retrieved context. Outputs True (ships immediately) or False (triggers System-2). No latency on success.

hallucination detection · faithfulness scoring
03

System-2 — Deep Search

On False verdict, the FDL engine lowers similarity thresholds, increases k, and activates GraphRAG relationship traversal to locate ground truth. A corrected answer is generated.

graph traversal · low-threshold retrieval
04

Failure Note Injection

A targeted correction is synthesized: "[CORRECTION]: agent claimed X — document states Y." Embedded at strength 1.0 as category=failure_note — permanently becomes the #1 result for that query class.

strength=1.0 · category=failure_note
05

Reinforcement or Decay

Successful retrievals reset last_accessed to now() and boost strength. Unused memories decay until they cross the suppression threshold and vanish from search — keeping retrieval fast and relevant.

spaced repetition · threshold pruning
Memory Metabolism

The mathematics of forgetting

Every memory node carries a living strength value. The system actively digests useful knowledge and excretes stale data based on actual field utility.

Ebbinghaus Decay Formula
S(t) = S₀ × e−λt
Adaptive Variant — Memory Worth + Context Relevance
M = sim(m,q) · (α·MW + γ·reward) · e−λt
S(t)
Current memory strength at time t
S₀
Initial or previous strength value
λ
Decay rate — 0.01 critical, 0.05 standard, 0.1 failures
t
Days elapsed since last_accessed
MW
Memory Worth — success/failure co-occurrence ratio
0.17
Suppression threshold — below this, excluded from search

FDL logs failures and stores failure records. Repeated errors surface failure memories, prompting correction.

TC-21CRITICAL

prune() every 5 turns with λ=0.02 preferences, λ=0.10 failures suppresses stale memories vs plain RAG.

TC-07DECAY
Decay Reduces Score

After 30 days at decay_rate=0.1, strength ≈ e^(−3) ≈ 0.05. Score must drop ≥50% vs t=0.

TC-08RETRIEVAL
Suppressed Excluded

Memories with suppressed=True must not appear in any retrieval, even on exact query match.

TC-12FDL
Failure Creates Memory

Each failure auto-stores a Memory with category="failure". Zero stored → one new failure record.

TC-17PRUNING
user_preference Protected

Preference memories exempt from worth-based pruning even at 0% success rate — permanently active.

TC-22E2E
Accuracy Ordering

Enhanced > Simple RAG > No Memory. Memory always helps; FDL makes enhanced marginally better.

TC-29SERIAL
JSON Output Valid

Output must contain ["no_memory","simple_rag","enhanced"] each with accuracy, error_repetition, memory_size_kb — all numeric.
